###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: Spring 없이 순수 자바로 H2를 쓰는 흐름과 JDBC 전반을 학습
  - H2 연결: Maven/Gradle 의존성 추가 후 `DriverManager.getConnection()`으로 직접 연결. 연결 URL 종류를 구분 — 파일 기반(`jdbc:h2:./data/testdb`), 인메모리(`jdbc:h2:mem:testdb`), 서버 모드(`jdbc:h2:tcp://...`)
  - JDBC 기초: JDBC는 DB마다 다른 구현을 표준 인터페이스로 통일한 API. 핵심 4요소 흐름은 `DriverManager`(연결 창구) → `Connection`(세션) → `Statement`/`PreparedStatement`(쿼리 실행) → `ResultSet`(결과)
  - 커넥션 풀 3종 비교: HikariCP(실무 표준, Spring과 무관한 순수 자바 라이브러리) / 직접 구현(BlockingQueue로 원리 이해용, 검증·타임아웃 없어 실전 부적합) / H2 내장 `JdbcConnectionPool`(별도 의존성 없이 H2에 포함된 심플한 풀, 기본 10개 커넥션·30초 타임아웃)
  - `DriverManager` vs `DataSource`: DriverManager는 매번 새 물리 연결을 만드는 저수준 API. DataSource는 커넥션을 "제공받는" 상위 인터페이스로 풀링을 강제하지 않음(HikariCP처럼 풀링하는 구현체도, Spring `DriverManagerDataSource`처럼 안 하는 구현체도 있음)
  - `DriverManager`의 현재 위상: 죽은 기술이 아님. HikariCP·JdbcConnectionPool 등 모든 풀 구현체가 내부적으로 여전히 DriverManager를 씀. 애플리케이션 코드에서 직접 호출하는 빈도만 줄었을 뿐(테스트·스크립트·학습용에는 지금도 사용)
  - JDBC 드라이버를 직접 만든다면: 관점이 뒤집힘 — DriverManager를 쓰는 게 아니라 DriverManager가 나를 호출하도록 구현/등록해야 함. 필수 구현 대상은 `Driver`, `Connection`, `Statement`, `ResultSet` 등이고 등록은 `META-INF/services/java.sql.Driver`(SPI) 또는 `registerDriver()`. 커넥션 풀링은 드라이버 책임이 아니라 그 위에 얹히는 별개 계층

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 커넥션 풀은 목적에 따라 선택 — 실무는 HikariCP, 원리 학습은 BlockingQueue 직접 구현, 별도 의존성을 피하고 싶은 간단한 경우엔 H2 내장 `JdbcConnectionPool`
- 그렇게 판단한 이유: 직접 구현은 검증·타임아웃이 없어 실전에 부적합하고, HikariCP는 Spring과 무관한 순수 자바 라이브러리라 순수 자바 환경에서도 그대로 쓸 수 있음. 학습·테스트 성격의 코드에서는 의존성 없이 즉시 쓸 수 있는 내장 풀이 트레이드오프상 합리적
- 고려했지만 선택하지 않은 대안: 커넥션 풀 없이 매 요청마다 `DriverManager`로 새 물리 연결 생성(비용·자원 관점에서 부적합), 모든 상황에 직접 구현 풀 사용(검증·타임아웃 부재로 실전 부적합)
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: `DataSource`가 풀링을 강제하지 않는 상위 인터페이스임을 이해하면서, "DataSource = 풀링"이라는 오해를 걷어냄. 실무에서 DataSource가 항상 풀링처럼 느껴지는 건 실제 쓰는 구현체 대부분이 HikariCP 같은 풀링 라이브러리이기 때문이라는 점을 정리. 또 JDBC 드라이버를 직접 만드는 관점에서 풀링이 드라이버 책임이 아닌 별개 계층임을 구분하게 됨

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: `No suitable driver` 오류로 H2 연결 실패
- 처음 추측한 원인: 드라이버 클래스 로딩 문제(연결 URL 오타 등)
- 원인을 확인하기 위해 시도한 방법: classpath에 h2.jar가 컴파일·실행 시점 모두 포함되어 있는지, 연결 URL 모드(tcp/파일/인메모리)가 실행 환경과 맞는지 확인
- 실제 원인: 컴파일 시점과 실행 시점 둘 다 classpath에 h2.jar가 필요한데 한쪽이 누락됨. tcp:// 모드는 서버가 먼저 떠 있어야 하는데 서버 없이 접속을 시도한 경우도 같은 증상으로 이어짐
- 해결 방법: 실행 시점 classpath에 h2.jar 포함을 보장. 서버를 별도로 띄우는 대신 `AUTO_SERVER=TRUE` 옵션으로 첫 접속 시 서버가 자동 기동되도록 해 더 간단하게 처리
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: `No suitable driver`가 뜨면 (1) 실행 시점 classpath에 드라이버 jar가 있는지 (2) tcp:// 등 서버 선행 기동이 필요한 URL 모드는 아닌지부터 확인
