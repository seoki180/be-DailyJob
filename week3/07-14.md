# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 정적 파일 서빙 → 서버 조립형 동적 렌더링으로 전면 교체. `static/*/index.html`을 제거하고 `templates/`(layout, header/guest·member, pages/)로 분해
  - 렌더링 책임을 3계층으로 분리: `PageController`(로그인 확인·라우팅·정적파일 서빙) / `PageService`(세션별 article·user 조회 및 분기 선택) / `TemplateBuilder`(데이터로 완전한 HTML 조립, Builder 패턴)
  - 게시판: 아티클 조회/렌더링(articleId를 queryString으로 강제 전송, 비로그인도 동일 조회), 이전글/다음글 네비게이션(글 존재 여부 확인 후 footer 동적 부착, 없으면 이동 불가·nav 위치는 유지), 댓글 작성 API(`/user/comment`), 댓글 3개 초과 시 확장 버튼·없으면 안내 문구, 좋아요(로그아웃 상태에서도 작동·클릭 수 카운트·영속화)
  - 파일 업로드(Multipart): `MultipartParser` + `UploadStorage` 구현(`HttpParser` multipart 처리 포함), 프로필 사진 업로드(`POST /user/profile`, 실제 저장 후 경로 렌더링·프로필 원 안에 맞게 표시), 아티클 작성 시 사진+본문 함께 업로드, 댓글에 작성자 프로필 사진 노출, upload 폴더 gitignore 처리
  - 버그/보안: 정적파일 서빙 취약점 제거(경로에 index.html 직접 넣어 인증 없이 접근하던 문제 차단), HttpParser 바디 파싱을 char→byte 기준으로 수정, nav href의 main 접두사로 인한 비로그인 게시물 탐색 문제 해결
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 어제 미완료였던 "page마다 나누어 렌더링하는 로직"을 계층 분리로 마무리함

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 동적 HTML 서빙을 하나의 메서드로 처리하려던 방식을 버리고 `PageController - PageService - TemplateBuilder` 3계층으로 책임을 분리
- 그렇게 판단한 이유: 로그인/비로그인 구분, 하단 nav 처리 등 분기가 많아 단일 메서드로는 복잡성이 오히려 증대되고 방향성을 잃었기 때문. 각 계층이 최소한의 책임만 지도록 하면 특정 부분 수정 시 다른 부분으로의 전파를 최소화할 수 있다고 판단
- 고려했지만 선택하지 않은 대안: 기존처럼 경로-폴더명을 매칭해 index.html을 서빙하는 방식(동적 처리 불가), 단일 메서드로 모든 분기 처리(복잡성 폭증)
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: HTML 생성 책임이 TemplateBuilder에 집중되어 헤더·바디 교체 지점이 명확해짐. 인증 없이 index.html에 직접 접근하던 서빙 취약점도 로직 재설계로 함께 차단됨

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 비로그인 상태에서 게시물 이전/다음글 탐색이 정상 동작하지 않음(로그인/비로그인 중 한쪽만 대응)
- 처음 추측한 원인: 세션 분기 로직 자체의 문제
- 원인을 확인하기 위해 시도한 방법: nav의 href 경로와 실제 라우팅 경로를 대조
- 실제 원인: nav href에 `main` 접두사가 붙어 있어 로그인/비로그인 중 한쪽 경로에만 매칭됨
- 해결 방법: href의 main 접두사 처리를 수정해 양쪽 상태 모두 대응하도록 변경. 추가로 HttpParser 바디를 char가 아닌 byte 기준으로 읽도록 수정(multipart 파싱 정확성 확보)
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 링크/라우팅이 상태별로 갈릴 때는 href 경로 규칙이 모든 상태에서 일관된지 먼저 확인

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 복잡성이 커진 단일 메서드 방식을 고집하지 않고, 페이지별로 필요한 데이터·메서드를 다시 정리해 계층/책임을 분리
- 그 접근이 도움이 된 이유: 각 계층이 최소 책임만 지니 수정 시 영향 범위가 좁아지고, 렌더링 흐름(라우팅→데이터 조회→HTML 조립)이 눈에 보이게 정리됨
- 다음에도 반복하고 싶은 점: 한 메서드가 비대해지고 방향성을 잃을 때, 무리해서 밀어붙이지 않고 책임 단위로 다시 쪼개는 판단

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 정적파일 서빙 취약점(인증 없이 index.html 직접 접근)을 설계 초기에 잡지 못하고, 렌더링 로직을 재설계하는 과정에서야 발견해 제거함
- 그렇게 된 원인: 처음 서빙 로직을 "경로에 맞는 파일 제공"이라는 기능 관점으로만 보고 접근 인증 관점을 함께 고려하지 않음
- 다음에는 다르게 해볼 구체적인 행동: 정적/동적 리소스 서빙을 구현할 때 "누가 접근 가능한가"를 기능 구현과 동시에 점검하는 것을 체크리스트로 둠
