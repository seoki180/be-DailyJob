###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 도메인 설계 관련 주요 의사결정 확정.
    - **결제**: PortOne 테스트 연동으로 카드 결제 진행하기로 결정. 선결제 원칙에 따라 결제가 항상 배송주문보다 선행되므로, **배송주문 테이블이 결제 FK를 보유**하는 구조로 확정. 결제 : 배송주문 = 1:1 (쿠폰/할인 도입 시에도 결제 행은 하나만).
    - **인증(대면/비대면)**: 테이블을 나누지 않고 하나의 배송 인증 테이블에 구분 컬럼(속성)을 두기로 결정. (한 주문에는 항상 인증 여부가 붙기 때문)
    - **전자 서명**: `signature_pad`(HTML5 canvas) 라이브러리로 웹뷰에서 그리고 바로 SVG로 추출하는 방식 채택. 단순 SVG 저장은 위변조 문제로 효력 인증이 어려울 수 있어, 타임스탬프·IP·접속 기기 정보를 백엔드에서 함께 기록하기로 함.
    - **주소 이력 / 가격**: 부르미가 주소를 중간에 수정할 때 예전 이력을 남기고, 가격은 참조가 아니라 literal hard copy로 주문에 박아두기로 결정.
    - **취소 정책**: 픽업 전/후 × 드리미/부르미 4가지 케이스별 취소·패널티·재매칭 규칙 정리. 배송 지연은 웬만하면 취소 불가, 예상시간의 2~3배를 넘으면 취소 프롬프트 노출.
    - **탈퇴 악용 방지**: soft delete 대신 '탈퇴 대기 상태'를 두고 일정 시간 경과 후 최종 탈퇴 처리.
    - **UI 플랫폼**: 모바일 기준으로 **웹사이트 + Capacitor** 채택.
    - **오늘의 핵심 결정**: 업체 인계 과정은 현실적으로 구현이 힘들어, 전문 업체로 넘기는 방향으로 스코프를 축소.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 콜이 안 잡힐 때 요금 인상(기본요금 + 거리 × 단가) + 팝업 알림/사용자 선택 로직 → **나중으로 미룸**.
  - 전자 서명의 법적 효력 관련 리서치(TODO).
  - 무게 검증 방식(현재는 '양심에 맡기기'로 임시 결론).

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 선결제 원칙을 데이터 모델에 반영해, **배송주문 테이블이 결제 FK를 소유**하도록 방향을 정함.
- 그렇게 판단한 이유:
  - "결제 → 배송주문 / 배송주문 → 결제" 중 어느 쪽이냐는 질문에서, 결제는 항상 선행되어야 하는 전제이므로 배송주문이 결제를 참조하는 것이 흐름상 자연스럽고 정합성 검증도 단순해짐.
- 고려했지만 선택하지 않은 대안:
  - 배송주문 생성 후 결제를 붙이는 방식(결제가 배송주문 FK 보유) → 결제 없는 주문 상태가 존재할 수 있어 선결제 원칙과 충돌.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - "결제 없는 주문"이라는 잘못된 상태 자체가 생기지 않아 검증 분기가 줄고, 1:1 제약으로 쿠폰/할인 확장 시에도 결제 행이 하나로 유지되어 정산·환불 로직이 단순해짐.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 업체 간 '인계' 과정을 어떻게 모델링/구현할지가 명확하지 않았고, 실시간 위치·상태 동기화까지 얽혀 스코프가 계속 커짐.
- 처음 추측한 원인:
  - 인계 테이블을 어떻게 분할하느냐의 설계 문제라고 생각.
- 원인을 확인하기 위해 시도한 방법:
  - 웹사이트 기반에서 GPS/실시간 반영을 검토 → 화면이 꺼지면 위치 추적이 끊기는 등 현실적 제약을 확인.
- 실제 원인:
  - 설계 문제가 아니라, 인계 자체가 주어진 시간·자원 안에서 감당하기 어려운 범위의 문제였음.
- 해결 방법:
  - 인계 과정을 직접 구현하지 않고 전문 업체로 넘기는 것으로 스코프를 축소.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - "이걸 우리가 직접 해야 하는가"를 설계보다 먼저 질문할 것. 구현 난이도가 아니라 스코프 자체를 먼저 검증.
