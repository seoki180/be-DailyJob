# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: user 회원가입 라우터 GET → POST 변경, 쿼리스트링 파싱 → body 파싱으로 변경 및 공통 parser로 함수명 통일, responseHandler → responseBuilder 구조 변경, httpStatusCode enum 생성, ThreadPool NPE 원인 규명 및 HttpParser null 체크 수정
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 쿠키로 세션 영속화 및 DB 대조

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: responseHandler의 정형화된 메서드 구조를 responseBuilder 패턴으로 전환
- 그렇게 판단한 이유: 기존 responseHandler는 200, 302, 404 등 완성된 응답만 반환해 새로운 형태가 필요할 때마다 메서드를 추가해야 했음. builder 패턴으로 status code, header, body를 유동적으로 조합할 수 있도록 변경
- 고려했지만 선택하지 않은 대안: responseHandler에 오버로딩 메서드를 계속 추가하는 방식
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 향후 리다이렉트, 커스텀 에러 응답 등 다양한 응답 형태를 단일 builder로 처리할 수 있게 되어 유지보수성 향상

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: ThreadPool 사용 중 특정 상황에서 NullPointerException 발생
- 처음 추측한 원인: ThreadPool을 의도적으로 적게 설정했을 때 발생하여, 특정 스레드가 Connection을 먼저 끊으면서 path가 null이 되는 것으로 추측
- 원인을 확인하기 위해 시도한 방법: Apache Benchmark로 Connection 수와 Request 수를 조정해가며 재현 테스트
- 실제 원인: ThreadPool 자체 문제가 아닌 Apache Benchmark의 동작 방식이 원인. ab는 HTTP/1.1에서 요청 라인 앞에 선행 `\r\n`을 전송하는 경우가 있음(RFC 7230 §3.5). 이로 인해 `readLine()`이 `""`를 반환하고 if 블록이 스킵되면서 path = null 상태로 Http 객체가 생성됨
- 해결: ThreadPool 로직은 수정하지 않고, HttpParser에서 `readLine()` 결과가 null 또는 빈 문자열일 경우 오류를 반환하도록 수정

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: ThreadPool을 수정하지 않고 실제 문제 발생 지점인 HttpParser만 정확히 타겟해 수정한 것
- 그 접근이 도움이 된 이유: RFC 문서로 ab의 동작을 검증해 추측이 아닌 근거 기반으로 원인을 확정했고, 변경 범위를 최소화해 사이드 이펙트 없이 수정 가능했음
- 다음에도 반복하고 싶은 점: 수정 전 스펙 문서(RFC 등)로 외부 도구의 동작을 먼저 확인하는 것

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 어제(07-03)부터 이어진 NPE 문제를 오늘에야 해결 — 처음부터 HTTP 파싱 단계를 의심했다면 더 빨리 해결 가능했을 것
- 그렇게 된 원인: Thread 레벨에 집중하느라 그보다 앞선 파싱 단계를 늦게 확인함
