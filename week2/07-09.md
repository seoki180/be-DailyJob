###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: WebServer를 Composition Root로 삼아 모든 인스턴스를 한 곳에서 생성하고 주입
- 그렇게 판단한 이유: AuthService 도입 후 각 Controller가 `new MemorySessionRepository()`를 직접 호출하면서 저장소 인스턴스가 Controller마다 따로 생성되는 문제가 드러났다. 로그인 후 페이지 이동 시 세션을 찾지 못하는 버그가 실제로 발생했고, 근본 원인이 인스턴스 분산임을 확인했다.
- 고려했지만 선택하지 않은 대안: 각 Controller가 직접 의존성을 생성하는 기존 방식 유지
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 세션 버그가 해결됐고, 의존성 그래프가 WebServer 한 곳에 집중되어 추후 DI 컨테이너 도입이나 테스트 시 교체가 용이해짐

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 로그인 후 페이지 이동 시 세션을 찾지 못해 인증이 풀리는 버그
- 처음 추측한 원인: 세션 저장/조회 로직 오류
- 원인을 확인하기 위해 시도한 방법: AuthService 분리 후 SessionRepository 참조 경로 추적
- 실제 원인: Controller마다 `new MemorySessionRepository()`를 직접 호출해 인스턴스가 분산 생성됨 — 저장한 세션과 조회하는 세션 저장소가 서로 다른 객체였음
- 해결 방법: WebServer(Composition Root)에서 SessionRepository 인스턴스를 단 한 번 생성하고 AuthService → Controller 순으로 주입
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 상태를 공유해야 하는 객체(Repository 등)가 여러 곳에서 new로 생성되고 있지 않은지 인스턴스 생성 지점부터 확인
