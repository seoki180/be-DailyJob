###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - 로그인 쿠키 생성을 UserService에서 RequestHandler로 옮기면서 관심사 분리. 로그인 실패 케이스도 커스텀 Exception으로 처리하도록 바꿨다
  - HttpCookie에 `toString()` 달아서 ResponseBuilder를 건드리지 않아도 쿠키 헤더가 나가게 했고, `HttpOnly`, `SameSite` 스펙도 챙겼다
  - 회원가입할 때 이미 있는 아이디면 예외 터지도록 중복 검사 추가
  - Router switch 구문을 인터페이스 기반으로 리팩터링해서 라우터 추가가 좀 더 자연스러워졌다
  - Repository에 interface 씌워서 UserService가 구체 구현체에 직접 의존하지 않도록 결합도 낮췄다
  - MemoryDatabase 내부 저장소를 `ConcurrentHashMap`으로 교체
  - SessionData 만들고 DB-Repository 패턴 입혀서 로그인 세션 영속화
  - `/user/logout` API 완성 — 서버에서 세션 지우고 클라이언트 쿠키도 만료시킨다
  - 프론트를 form 방식에서 JS fetch로 전환하고, 요청 실패 시 토스트 뜨도록 처리
  - 프론트 로그아웃 버튼 연동
  - 회원가입 body에 값이 빠져 있어도 가입이 되던 버그 수정
  - `Database` → `MemoryDatabase` 이름 정리, 토스트 등장 시간 0.5초 줄임

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: Repository 레이어를 도입해 Service가 DB 접근 방식을 몰라도 되도록 책임을 분리한 것
- 그렇게 판단한 이유: 처음엔 "이 정도 규모에 Repository까지 필요한가?" 싶었는데, 세션 데이터가 추가되면서 Service가 도메인 로직이랑 DB 접근 로직을 동시에 떠안기 시작했다. 이대로 가면 저장소가 바뀔 때 Service도 같이 열어야 하는 상황이 뻔히 보여서 분리하기로 했다
- 고려했지만 선택하지 않은 대안: Service에서 계속 직접 DB 호출 — 단순하긴 하지만 저장할 데이터가 늘어날수록 Service가 뚱뚱해지는 문제가 있었다
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: java식으로 코드를 짜는것에 익숙하지 않아 interface를 통한 구현에 익숙하지 않았다. 하지만 Object 책을 읽으머 추상화와 구현의 중요성에 대해 얕게나마 깨달을수 있었고 이를 점차 적용하는 방향으로 구현을 시작했다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 서버의 응답을 클라이언트에서 어떻게 처리할지 고민이 되었다. 완전한 SSR로 구현하기 위해서는 모든 html을 서버에서 생성하고 클라이언트는 이를 로딩하고 redirect하는 로직으로 구현해야하는데 그렇다면 alert같은 로직을 구현하기 애매했다.
- 해결: rest 형식의 구현을 도입해 서버는 상태를 status code로 응답하기만 하고 이에 대한 로직 구현은 클라이언트가 전담하는방향으로 로직을 구현했다.
